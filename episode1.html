1.2 How Computers Communicate

graph LR A[Application Layer
Your Browser] --> B[Transport Layer
TCP/UDP] B --> C[Network Layer
IP Addressing] C --> D[Data Link Layer
MAC Address] D --> E[Physical Layer
Cables/WiFi] E --> F[Router/Switch] --> G[ISP Network] --> H[Destination Server]

📦 Data Journey Packet Structure:

┌─────────────────────────────┐
│ Application Data (HTTP)     │ ← What you want to send
├─────────────────────────────┤
│ TCP Header                  │ ← Port numbers, sequencing
├─────────────────────────────┤
│ IP Header                   │ ← Source/Dest IP addresses
├─────────────────────────────┤
│ Ethernet Header             │ ← MAC addresses (local network)
├─────────────────────────────┤
│ Physical Signal (0s and 1s) │ ← Electrical/Optical transmission
└─────────────────────────────┘

1.3 DNS Resolution Flow

sequenceDiagram participant U as User participant B as Browser Cache participant OS as OS/DNS Client participant R as Recursive DNS Resolver participant TLD as TLD Nameserver (.com) participant A as Authoritative Nameserver U->>B: Type "example.com" B->>B: Check cache alt Cache Hit B-->>U: Return IP immediately else Cache Miss B->>OS: Request DNS lookup OS->>R: Query recursive resolver R->>TLD: Ask for .com nameserver TLD-->>R: Return authoritative NS R->>A: Query for example.com A-->>R: Return IP: 93.184.216.34 R-->>OS: Cache & return IP OS-->>B: Return IP B->>A: Connect to IP end
💡 Key Insight: MAC addresses never leave your local network. Routers strip and replace them at each hop using IP addresses for long-distance routing.

3. Internet Protocols: TCP vs UDP

3.1 TCP 3-Way Handshake

Client                          Server
   |--- SYN (seq=x) ------------->|  1. Request connection
   |<-- SYN-ACK (seq=y, ack=x+1) --|  2. Acknowledge + request
   |--- ACK (ack=y+1) ------------>|  3. Confirm
   |====== Connection Established ======|
   |--- Data Packets ------------->|
   |<-- Acknowledgments -----------|

3.2 TCP vs UDP

FeatureTCPUDP
ConnectionConnection-oriented (handshake)Connectionless
Reliability✅ Guaranteed delivery❌ No guarantees
Ordering✅ Packets arrive in order❌ May arrive out of order
Speed🐌 Slower (reliability overhead)⚡ Faster (minimal overhead)
Use CasesWeb, Email, FilesVideo calls, Gaming, DNS
📦 TCP = Registered Mail (tracked, reliable)
📦 UDP = Postcard (fast, may get lost)

4. HTTP & HTTPS Deep Dive

4.1 HTTP Status Codes

200 OK          → Success
201 Created     → Resource created (POST)
301/302         → Redirects
400 Bad Request → Client sent invalid data
401 Unauthorized→ Login required
403 Forbidden   → No permission
404 Not Found   → Resource missing
429 Too Many    → Rate limited
500 Server Error→ Bug on server

4.2 HTTPS: Security Flow

sequenceDiagram participant C as Client participant S as Server participant CA as Certificate Authority Note over C,S: 1. Client Hello C->>S: Supported TLS versions, cipher suites Note over C,S: 2. Server Hello + Certificate S->>C: Chosen TLS version, certificate C->>CA: Verify certificate signature CA-->>C: Certificate valid ✅ Note over C,S: 3. Key Exchange C->>S: Client public key + encrypted pre-master S->>C: Acknowledgment Note over C,S: 4. Generate Session Keys (Symmetric) C->>C: Derive keys S->>S: Derive same keys Note over C,S: 5. Encrypted Communication C->>S: 🔐 Encrypted HTTP request S->>C: 🔐 Encrypted HTTP response
🔐 Critical Headers:
Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: default-src 'self'

<form id="signup-form" novalidate>
  <fieldset>
    <legend>Account Information</legend>
    <label for="email">Email</label>

✅ Accessibility Checklist:
• One <h1> per page
• All inputs have labels
• Color contrast ≥ 4.5:1
• Focus indicators visible
• Dynamic updates use aria-live
